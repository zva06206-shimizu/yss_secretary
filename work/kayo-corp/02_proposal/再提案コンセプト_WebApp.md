# 再提案コンセプト — BOM×三菱支給 照合・粗利分析 Web App

作成：2026-07-01（初版・コンセプト）

## 1. なぜ再提案か（背景）

- カヨウは**受注の大半が三菱電機、部品の多くが三菱からの支給品**。原価は三菱向け製品ごとの Excel「部品構成ファイル（多階層BOM／見積）」で管理。
- 前回2月案件は「診断・課題抽出・改善案・提案資料」まで（実装は契約外）。
- **止まった理由＝BOM（部品構成ファイル）と三菱の支給品明細の"照合"をシステム化するのが技術的に困難だったから**。SQLite＋Power Query 路線で試作（`04_system/202602/` の参考スクリプト群）まで進んだが、突合が壁になった。

## 2. 再提案で作りたいもの（Yutaka意向）

**Web App** として、以下を実現する：

1. **BOMファイルと三菱支給ファイルを照合**（部品コードで突合）。
2. **粗利**を可視化（売上−原価。原価＝BOMの材料費＋加工費＋管理費）。
3. **支給部品比率**を可視化（支給品／全部品、金額・点数ベース）。
4. **「支給→自社調達」切替シミュレーション**：ある支給部品を自社調達に切り替えた場合、粗利がどう改善するかを試算。

→ 位置づけ＝**"原価と調達戦略の見える化ツール"**。三菱依存下で利益を守る意思決定を支える。

## 3. クロエの反証（ここが生命線）

- **前回を殺したのは"照合"。Web化はUI・提供形態の改善であって、照合の困難そのものは解けない。**
- 突合を難しくしている要因（仕様原文＋現場反証より）：
  - BOMのレイアウトが **A/B 2パターン・列構成が不揃い・行数可変・工程コード行が可変**。
  - **親が見つからない単独シート**が存在し得る。
  - **生産終了品**が混在（支給明細の部品コードからたどれない）。
  - 支給明細は SC始まりシート・12ヶ月累積など独自構造。
- → **照合ロジックを先に PoC で解いてから提案する**。UIから入ると同じ壁で二度死ぬ。

### 3.1 なぜ前回失敗し、今回は勝てるのか（技術的核心）

- 【現場反証 2026-07-01】**BOMの列構成が同じでないため、Power Query での抽出が非常に困難だった**。これが前回 SQLite＋Power Query 路線の技術的敗因の本体。
- **Power Query は「列が揃った表」を前提とする道具**＝不定形なブロック構造（材料費/加工費/管理費、可変行、A/B列違い）に弱い。前回の失敗は路線選択の問題であって、要件が不可能なわけではない。
- **今回の勝ち筋＝Power Queryをやめ、柔軟なパーサ（Python）で"ブロック検出→正規化"してから構造化データにする**。セル位置(E13=親コード等)・ブロック見出し・A/B判定をコードで吸収すれば、不定形Excelを吸い込める。前回スクリプト資産(`04_system/202602/`)はこのパーサの土台になり得る。
- → **Web App の価値の8割はUIではなく、この"不定形BOMを飲み込むパーサ＋照合エンジン"**。ここがPoCで最初に潰すべき一点。

### 3.2 版管理の設計原則（並行生産への対応）

- 【現場反証 2026-07-01】**機械判定で最新と思った版が、実は旧版と並行生産されていることがある**。＝「最大改訂＝有効版」は誤り。
- 設計原則：**有効版は"ファイル名の最大改訂"ではなく"支給明細に実績がある版"で判定する**。並行生産も支給実績で自然に捕捉できる。
- 機械で決めきれない領域（同一改訂・日付違い28%、破損、並行生産の最終判断）は**工場長レビュー**に回す「機械判定＋例外レビュー」の二層で。詳細 [../01_business/BOM版管理_問題と分析.md](../01_business/BOM版管理_問題と分析.md)。

### 3.3 その他の現実確認

- **三菱支給品を自社調達に切り替えられるかは商流上の制約がある**（支給は三菱の価格・品質・管理方針に紐づく）。シミュレーションは「交渉・計画の材料」としては強いが、「自由に切替可能」を前提に語ると足元をすくわれる。**"改善余地の可視化"**として位置づけるのが安全。

## 4. 進め方（提案・実装の順序案）

1. **PoC（照合の実現可能性検証）**：手元の実データ（部品構成9件＋三菱売上・材料支給明細）で、BOM×支給の突合が例外込みで安定するかを検証。前回スクリプト資産を土台に使えるか評価。
2. PoC結果 → **提案書**（何が見える化でき、粗利改善がいくら見込めるか＝会計士とROIを裏取り）。
3. 承認後に **Web App 実装**（照合エンジン＋粗利/比率/シミュレーションUI）。
4. **運用**（月次の支給明細更新への追随、再抽出フロー）。

## 5. 論点・未確認

- Web App の提供形態：社内スタンドアロン／クラウド／Excelアドイン相当。三菱データを扱うため**セキュリティ・データ保管場所**の要件確認が必須（YSSの中小セキュリティの強みが活きる）。
- 前回スクリプト（SQLite抽出）の到達点＝どこまで動いたか。照合が「一部できた／全くできない」かで再利用度が変わる。
- 粗利の計算に必要なデータが全て揃うか（売価＝三菱売上、原価＝BOM、支給単価＝支給明細）。3点が部品コードで繋がるかがPoCの核。
- スコープと金額：PoC（小額・実現性検証）→ 本実装（本体）の2段構えで、前回の「大きく提案して止まる」を避ける。
