## 2. 進め方の骨子 ― 3つの層で考える

| 層 | 目的 | 主な打ち手 | 社長関心との対応 |
|---|---|---|---|
| **守る（入れない・広げない）** | 侵入を防ぎ、入られても横展開させない | NWセグメント分離・ACL・UTM・拠点間の最小通信・ゼロトラスト的分離 | ランサム対策 |

この3層を、**現状把握 → 方針確定 → 詳細設計 → 伴走**と段階的に進めます（募集要項の「相談→設計→伴走」と同じ進め方）。

## 3. ランサムウェア対策の二段構え（経営目線の絵）

```
        ┌─────────────────────────────────────────┐
        │  ① 入れない・広げない（ネットワーク側）   │  ← 今回のNW刷新が土台
        │  ・拠点/工場/基幹DBをセグメント分離        │
        │  ・UTMで出入口を監視・遮断                │
        │  ・拠点間は必要最小限の通信だけ許可        │
        └─────────────────────────────────────────┘
                         ↓ もし破られても
        ┌─────────────────────────────────────────┐
        │  ② やられても戻せる（バックアップ側=BCP） │  ← 社長指示①
        │  ・基幹(VB+SQL)を定期バックアップ          │
        │  ・書き換えられない/切り離した保管         │
        │  ・「復元できる」を演習で証明              │
        └─────────────────────────────────────────┘
```

**ポイント**：多くの会社は「バックアップは取っている」が「**戻せるか試したことがない**」。ランサムは保管中のバックアップごと暗号化しに来るため、**①切り離し（イミュータブル/オフライン）②復元演習**までやって初めて“BCP”と呼べます。ここを確実にします。

---

## 4. 「復元できるバックアップ」の考え方（BCP）

- **3-2-1の原則**：3つの複製・2種類の媒体・1つは隔離（オフライン/別拠点/クラウド）。
- **書き換え不能保管**（イミュータブル／オフライン）でランサムに巻き込まれない退避先を確保。
- **復元目標を決める**：止まってどれくらいで戻すか（RTO）、どの時点まで戻すか（RPO）を業務と相談して数値で握る。
- **復元演習**：年◯回、実際に戻して時間と手順を検証。手順書を残し、属人化させない。

> 基幹がVB＋SQLの社内開発という点は、**SQLのバックアップ方式と、アプリ側（VB）含めた復旧手順**を一体で設計する必要があります。ここはこたけさんと一緒に詰めたい領域です。
