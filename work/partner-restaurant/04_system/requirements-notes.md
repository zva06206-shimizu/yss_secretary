# Dining Bar Jakarta 運営の仕組み・構想メモ（2026-07-02）

Yutaka の検討メモを記録。まだ「考えること」の段階＝要件の種。決定ではない。

## 1. 雇用形態（スタッフ）

- **ベース賃金＋売上歩合**。
- 休み：**日曜・月曜**。
- 勤務時間：**20:00〜23:00**。
- 給与支払：**月2回**。

> 検討ポイント（クロエ）：勤務20〜23時＝営業時間もこの枠か（接待/深夜酒類の許可判断に直結 → [staffing-and-license.md](staffing-and-license.md)）。歩合率・ベース額・締め日/支払日は未定＝繋ぎ止め条件として早めに固める。

## 2. 源泉徴収の仕組み

- **売上を入力したら源泉徴収額が出る**ツールが欲しい。
- 実装案：**Excelシート または Web App**。

> 検討ポイント（クロエ）：ホステス等への報酬は源泉徴収の計算方法が特殊（1回払で「（支払額−控除額）×10.21%」等、給与とは別体系のことがある）。**「ベース賃金＝給与所得の源泉」なのか「歩合＝報酬の源泉」なのか、雇用/業務委託の別**で計算が変わる。税理士に区分を確認してから計算式を組む。レジ(3)と連動させれば売上入力→歩合→源泉まで一気通貫にできる。

### 検討ポイント（クロエ・反証役）

- **8月完成は攻めた日程**。オープンが8月お盆明け〜9月なので開店に間に合わせたい意図は妥当。ただしフル自作Web Appは要件が多い（多言語・プリンタ連携・顧客管理・歩合・CSV）。
  - **現実解の比較を先に**：①**エアレジ等の既製POS＋CSV連携**で開店を確実にし、②歩合・源泉・顧客の“痒い所”だけ自作補助ツールで足す、というハイブリッドも選択肢。フル自作は開店後に育てる手もある。
  - まず**MVP＝「iPad伝票→会計→mC-Print3レシート→スタッフ紐付け→CSV出力」**に絞れば8月に届きうる。多言語・顧客管理・源泉連動は第2段でよい。
- **mC-Print3連携**：iPad＋Web App＋mC-Print3は、Star公式のWebPRNT/SDK経由で印刷可。自作WebAppでも実装可能だが、印刷は最初の技術リスク＝早めに疎通確認したい。
- **MFクラウド取込**：CSVの列設計を先にMF側の受け口に合わせると後が楽（勘定科目・税区分・日付・金額）。
- 会計としてのレジ＝**現金・釣銭・売上日報・締め**まで要るか要確認（歩合と源泉だけなら簡易でよいが、店の会計なら日次締めが要る）。

## エアレジ（Airレジ）調査結果（2026-07-02・公式FAQベース）

**結論：無料で、要件の大半（iPad／mC-Print3／顧客登録／自宅閲覧／CSV→MF）を即満たす。ただし“ナイトの歩合”に必須の「品目×スタッフ紐付け」が弱い＝ここだけ自作で補うのが現実解。**

| 要件 | エアレジで可否 | 補足 |
|---|---|---|
| 料金 | ◎ **初期0円・月額0円・全機能無料** | POSアプリとして無料。決済(Airペイ)は別 |
| R1 iPad伝票・会計・レシート印刷 | ◎ 可 | iPad/iPhone専用アプリ |
| R1 mC-Print3印刷 | ○ 可（**要確認**） | 対応は**mC-Print3の"MCP31"系（例MCP31LB）に限る**。Yutaka保有機がこの型番か要確認 |
| R2 メニュー英語・インドネシア語 | △ 部分的 | 商品名は自由入力＝英・尼語で**登録**は可。**画面の言語切替UI**は標準機能として明確でない。レシート多言語フォントは日/英/中/韓（尼語=ラテン文字なので英ANKで印字可） |
| R3 伝票にスタッフ名（歩合） | **▲ ここが弱点** | 会計明細CSVに「**レジ担当者名**」項目あり＝**会計（卓）単位の担当者別売上**は出せる。だが「このボトルはナンシー、この女子ドリンクはＡ子」という**品目ごとに別スタッフを紐付ける**運用は標準では想定薄。ナイトの歩合の肝がここ |
| R4 常連顧客の登録 | ◎ 可 | 名前・電話等を登録し会計に紐付け→来店/会計履歴。※**顧客データのCSV出力は不可**、DM送信も不可 |
| R5 自宅から管理画面 | ◎ 可 | バックオフィスはPC/ブラウザ＝自宅から売上確認可 |
| R6 CSV出力→MFクラウド | ◎ 可 | 売上集計・**会計明細**・商品別売上など多数CSV出力。MFクラウド会計と**連携あり（ただし取込は“日別売上”単位＝明細単位ではない）** |
| R7 歩合・源泉の算出 | ✕ 標準機能なし | エアレジは計算しない→**CSVを外部ツール(自作/Excel)に食わせて計算**する形 |

### この事実からの現実解（クロエ推奨）

- **フルスクラッチのレジを8月に間に合わせる必要はない**。エアレジで①iPad会計②mC-Print3印刷③顧客登録④自宅閲覧⑤CSVは**無料で即日そろう**＝開店を落とさない。
- **自作すべきは“ナイトの歩合”だけ**：エアレジの弱点＝品目×スタッフ紐付け＆歩合/源泉計算(R3の細粒度・R7)。ここを、**エアレジ会計明細CSVを取り込んで「スタッフ別売上→歩合→源泉」を出す軽い補助ツール**（Excel or 小さなWeb App）で足す。
- **割り切り案**：エアレジは「**卓＝担当（例：ナンシー卓）**」の粒度で担当者を付ける運用にすれば、品目単位に拘らず歩合の土台になる。指名ボトル等の例外だけ手入力補正。これで自作範囲がさらに小さくなる。
- → **ハイブリッド（エアレジ＋自作は歩合/源泉だけ）が、8月開店・低リスク・低コストの最有力**。フル自作は開店後に育てる選択肢として温存。

**次アクション候補**：①保有mC-Print3の型番確認（MCP31系か）②エアレジで卓/担当運用が歩合に足るかデモ確認③会計明細CSVの列を見て歩合ツールの設計。

## 方針決定：エアレジ型 Web App を自作（統合運営システムへ拡大）（2026-07-02）

Yutaka の方針＝**エアレジのような Web App を自作**。単なるPOSでなく、**伝票→スタッフ別売上→歩合→源泉徴収→月2回給与、＋出退勤打刻**まで1つに統合する構想。

### 狙い＝データが1本の線でつながる（これが自作する理由）

```
出退勤打刻(iPad・名前選択)          伝票(誰の売上か=スタッフ紐付け)
        │                                    │
        └─ 労働時間 ─┐          ┌─ スタッフ別売上(源泉の元になる売上)
                     ↓          ↓
              ベース賃金 + 売上×歩合率 = 支給額
                          ↓
              源泉徴収(令和8年 月額表ロジック) → 差引支給額
                          ↓
                  月2回の給与支払いが自動で楽に
```

- **エアレジの弱点（品目×スタッフ紐付け・歩合・源泉が出せない）を、まさにここで解消する**＝自作の正当な理由。既製POSでは「伝票→誰の売上→歩合→源泉→給与」の一気通貫は作れない。
- 出退勤打刻を同じアプリに載せる＝**勤怠(ベース賃金の時間)と売上(歩合)が同一のスタッフマスタに乗る**＝二重入力なし。合理的。

### ★中核ロジックは既に完成＝源泉徴収Excel（[06_data/gensen_kyuyo_keisan_2026.xlsx](../06_data/gensen_kyuyo_keisan_2026.xlsx)）

Yutaka作成。**令和8年分 給与所得の源泉徴収税額表（月額表）に準拠**した計算シート。Web Appはこのロジックをそのまま移植すればよい（＝一番難しく間違えやすい税計算は検証済み）。

- 入力：支払月・スタッフ名・**売上金額・歩合率**（→支給額=売上×歩合、または支給額直接入力も可）・社保控除額・**税額表区分(甲/乙)**・**扶養親族数**。
- 計算：控除後給与 → **月額表を INDEX/MATCH 参照**して源泉所得税 → 差引支給額。甲欄(0〜7人)・乙欄の両方に対応。スタッフ別に支給額/源泉税/差引の月次集計あり。
- 出典＝国税庁 令和8年分 月額表（シート内に出典URL明記）。「賞与/日払い/丙欄/年末調整/住民税/雇用保険は対象外・最終確認は税理士」と本人が注記済み＝前提が明確。
- **設計インプットとして重要**：この列構成（甲/乙・扶養数・社保控除・歩合率）がそのままWeb Appの給与計算モジュールの仕様になる。

### 反証・要検討（クロエ）

1. **8月開店にスコープが膨らむ**：POS＋勤怠＋給与/源泉を一度にフル自作は重い。**開店を落とさないため段階リリース**を強く推奨：
   - **MVP（8月・開店必須）**：iPad伝票→会計→mC-Print3レシート→**伝票にスタッフ紐付け**→日次売上→CSV。まず「店が回る＋誰の売上か記録される」まで。
   - **第2段（開店後すぐ）**：スタッフ別売上集計→**歩合→源泉（Excelロジック移植）→月2回給与**。※開業直後の初回給与に間に合えばよい＝それまでは既存Excelで手計算できる保険がある。
   - **第3段**：出退勤打刻・常連顧客管理・多言語UI。
2. **雇用か業務委託かの区分**：Excelが**給与所得(月額表)ルート**＝**雇用契約(給与所得者)前提**。出退勤打刻・ベース賃金もこれと整合。だが雇用にすると**労働時間管理・最低賃金・(条件次第で)社会保険・労災**の義務が出る。→ **雇用で行く前提を税理士に確認**（[staffing-and-license.md](staffing-and-license.md)の区分確認と同じ論点）。甲欄は扶養控除等申告書の提出が必要＝掛け持ち者は乙欄。
3. **月2回払いの源泉単位**：月額表は本来「その月の給与」に対して用いる。**月2回に分けて払う場合の源泉計算の当て方**（合算か・各回か）は国税庁ルールを税理士に確認してから実装（過不足を避ける）。
4. **技術リスクの先出し**：mC-Print3印刷（Webから）＋iPad運用は最初に疎通確認。オフライン耐性（店の回線が落ちても会計できるか）も設計時に方針を決める。
