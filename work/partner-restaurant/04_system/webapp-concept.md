# Dining Bar Jakarta 店舗運営 Web App 構想書（統合版）

作成：2026-07-02 ／ 状態：構想（要件の種）。詳細・反証は [requirements-notes.md](requirements-notes.md)、業態/許可は [staffing-and-license.md](../00_context/staffing-and-license.md) 参照。

## 1. 目的（なぜ作るのか）

「エアレジのようなPOS」を自作し、**伝票・売上・スタッフ・歩合・源泉徴収・給与・出退勤を1つのアプリに統合**する。
狙いは、**伝票で拾った「誰の売上か」を起点に、歩合→源泉→月2回の給与計算まで自動で流す**こと。
既製POS（エアレジ等）は「伝票の品目×スタッフ紐付け→歩合→源泉→給与」の一気通貫ができない＝**そこが自作の正当な理由**。

## 2. 全体像（データが1本の線でつながる）

```
[出退勤打刻] iPadで名前を選んで打刻 ──► 労働時間 ─┐
                                                    ├─► 支給額 = ベース賃金 + (スタッフ別売上 × 歩合率)
[伝票/会計] 品目に担当スタッフを紐付け ─► 売上 ────┘            │
   （ボトル・女子ドリンク＝誰の売上か）                          ▼
                                          源泉徴収（令和8年 月額表ロジック＝Excel移植）
                                                                 │
                                                                 ▼
                                              差引支給額 → 月2回の給与支払いを自動集計
   ＋ 常連顧客登録 / 自宅から管理画面 / 売上CSV → MFクラウド
```

## 3. 機能一覧（モジュール別）

### A. POS・会計（店舗オペレーション）
- iPadで**伝票作成・会計・レシート印刷**（プリンタ＝保有の **mC-Print3**）
- メニュー登録・注文入力。**メニュー表示は英語＋インドネシア語**（できれば／最低限 商品名を多言語登録）
- **伝票の品目に担当スタッフを紐付け**（ボトル・女子のドリンク等）＝**誰の売上かを記録**（歩合と源泉の元データ）
- 会計方法（現金／必要なら後日カード）・レシート発行

### B. スタッフ・勤怠
- スタッフマスタ（名前・歩合率・税区分 甲/乙・扶養人数・ベース賃金/時給）
- **出退勤打刻**：iPadでアプリを開き**名前を選んで打刻**（出勤/退勤）→労働時間を記録

### C. 給与・源泉徴収（★中核・ロジック検証済）
- **スタッフ別売上**を集計（Aの伝票紐付けから）
- 支給額 ＝ ベース賃金 ＋ 売上 × 歩合率
- **源泉所得税を自動計算**：令和8年分 給与所得の源泉徴収税額表（月額表）準拠。甲/乙欄・扶養0〜7人対応。（ロジックは [gensen_kyuyo_keisan_2026.xlsx](../06_data/gensen_kyuyo_keisan_2026.xlsx) をそのまま移植）
- 差引支給額を算出し、**月2回の給与支払い**を集計（支払月ごと・スタッフごと）

### D. 顧客・CRM・データ連携
- **常連顧客の登録**（名前・連絡先・来店/会計履歴・※将来ボトルキープ）
- **顧客に担当スタッフを紐付け**（指名/担当）＝誰の常連か
- **スタッフ向けダッシュボード**：自分の担当顧客の**月ごとの来店数・売上**を**スタッフ本人に見せられる**（成績可視化＝指名文化のモチベ設計。※他人の数字は見せない）
- **メール・LINE取得時の店からの配信**：メール／LINEを取得できた顧客へ、統合アプリから**ステップメール（自動配信）**を送れる（来店促進・記念日DM等）
- **売上データCSV出力**（→ **マネーフォワードクラウド**へインポート）
- **管理画面は自宅からも確認**（クラウド前提の遠隔閲覧）

> 検討（クロエ）：①**配信は法対応が前提**＝広告メールは**特定電子メール法**でオプトイン取得・送信者表示・配信停止導線が必須。LINE配信は**LINE公式アカウント＋Messaging API**（無料通数に上限、超過は有料）。個人情報の取得同意・管理も要設計。②スタッフに数字を見せるのは効果的だが**自分の担当分のみ**に絞る。③ステップメール/LINEは**第3段**（開店必須ではない）。

## 4. 非機能・前提

| 項目 | 内容 |
|---|---|
| 端末 | iPad（店頭オペ）／PC・スマホ（自宅から管理画面） |
| プリンタ | mC-Print3（Webからの印刷疎通を最初に確認・型番も確認） |
| 稼働形態 | クラウド（自宅から閲覧）。店の回線断時の会計継続（オフライン耐性）は方針を決める |
| 言語 | 日本語＋英語＋インドネシア語（メニュー中心） |
| 会計連携 | MFクラウドのCSV取込フォーマットに列設計を合わせる |
| 対象外（当面） | 賞与・日雇丙欄・年末調整・住民税・雇用保険料の計算（Excel注記どおり） |

## 5. リリース計画（8月開店を落とさない段階リリース）

| 段階 | 範囲 | 目的 | 時期 |
|---|---|---|---|
| **MVP** | A：iPad伝票→会計→mC-Print3レシート→**スタッフ紐付け**→日次売上→CSV | 店が回る＋誰の売上か記録される | **8月・開店必須** |
| **第2段** | C：スタッフ別売上→歩合→源泉（Excel移植）→**月2回給与** | 給与計算を自動化（初回給与に間に合えば可・それまではExcelで手計算の保険あり） | 開店直後 |
| **第3段** | B：出退勤打刻／D：顧客×スタッフ紐付け・スタッフ向け成績ダッシュボード・**ステップメール/LINE配信**・多言語UIの拡充 | 運用を楽に・再来店を作る | 開店後 |

> フル機能を8月に一度に作るのはリスク。**MVPで開店を確実にし、給与・勤怠・顧客は段階で載せる**。既存Excelがあるので給与は手計算で繋げる＝第2段が多少ずれても実害小。

## 6. 着手前に決める/確認する（判断待ち）

1. **雇用 or 業務委託**：Excelは給与所得(月額表)前提＝**雇用**。出退勤・ベース賃金とも整合。雇用なら労働時間管理・最低賃金・(条件次第で)社保/労災の義務。→ **税理士に区分確認**。
2. **月2回払いの源泉の当て方**（合算か各回か）を国税庁ルールで確認してから実装。
3. **甲/乙の適用**：掛け持ちスタッフは乙欄（扶養控除等申告書は1社のみ甲）。
4. **技術**：mC-Print3のWeb印刷疎通、オフライン耐性の要否。
5. **作るか/ハイブリッドか**：フル自作 or 「エアレジ＋自作の歩合/源泉ツール」の最終判断（[requirements-notes.md](requirements-notes.md) 参照）。自作なら本構想でMVP設計へ。
