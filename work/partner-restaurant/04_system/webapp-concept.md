# Dining Bar Jakarta 店舗運営 Web App 構想書（統合版）

作成：2026-07-02 ／ 状態：構想（要件の種）。詳細・反証は [requirements-notes.md](requirements-notes.md)、業態/許可は [staffing-and-license.md](../00_context/staffing-and-license.md) 参照。

## 1. 目的（なぜ作るのか）

「エアレジのようなPOS」を自作し、**伝票・売上・スタッフ・歩合・源泉徴収・給与・出退勤を1つのアプリに統合**する。
狙いは、**伝票で拾った「誰の売上か」を起点に、歩合→源泉→月2回の給与計算まで自動で流す**こと。
既製POS（エアレジ等）は「伝票の品目×スタッフ紐付け→歩合→源泉→給与」の一気通貫ができない＝**そこが自作の正当な理由**。

## 2. 全体像（データが1本の線でつながる）

```
[出退勤打刻] iPadで名前を選んで打刻 ──► 労働時間 ─┐
                                                    ├─► 支給額 = ベース賃金 + (スタッフ別売上 × 歩合率)
[伝票/会計] 品目に担当スタッフを紐付け ─► 売上 ────┘            │
   （ボトル・女子ドリンク＝誰の売上か）                          ▼
                                          源泉徴収（令和8年 月額表ロジック＝Excel移植）
                                                                 │
                                                                 ▼
                                              差引支給額 → 月2回の給与支払いを自動集計
   ＋ 常連顧客登録 / 自宅から管理画面 / 売上CSV → MFクラウド
```

## 3. 機能一覧（モジュール別）

### A. POS・会計（店舗オペレーション）
- iPadで**伝票作成・会計・レシート印刷**（プリンタ＝保有の **mC-Print3**）
- メニュー登録・注文入力。**メニュー表示は英語＋インドネシア語**（できれば／最低限 商品名を多言語登録）
- **伝票の品目に担当スタッフを紐付け**（ボトル・女子のドリンク等）＝**誰の売上かを記録**（歩合と源泉の元データ）
- 会計方法（現金／必要なら後日カード）・レシート発行

### B. スタッフ・勤怠
- スタッフマスタ（名前・歩合率・税区分 甲/乙・扶養人数・ベース賃金/時給）
- **出退勤打刻**：iPadでアプリを開き**名前を選んで打刻**（出勤/退勤）→労働時間を記録

### C. 給与・源泉徴収（★中核・ロジック検証済）
- **スタッフ別売上**を集計（Aの伝票紐付けから）
- 支給額 ＝ ベース賃金 ＋ 売上 × 歩合率
- **源泉所得税を自動計算**：令和8年分 給与所得の源泉徴収税額表（月額表）準拠。甲/乙欄・扶養0〜7人対応。（ロジックは [gensen_kyuyo_keisan_2026.xlsx](../06_data/gensen_kyuyo_keisan_2026.xlsx) をそのまま移植）
- 差引支給額を算出し、**月2回の給与支払い**を集計（支払月ごと・スタッフごと）

### D. 顧客・CRM・データ連携
- **常連顧客の登録**（名前・連絡先・来店/会計履歴・※将来ボトルキープ）
- **顧客に担当スタッフを紐付け**（指名/担当）＝誰の常連か
- **スタッフ向けダッシュボード**：自分の担当顧客の**月ごとの来店数・売上**を**スタッフ本人に見せられる**（成績可視化＝指名文化のモチベ設計。※他人の数字は見せない）
- **メール・LINE取得時の店からの配信**：メール／LINEを取得できた顧客へ、統合アプリから**ステップメール（自動配信）**を送れる（来店促進・記念日DM等）
- **売上データCSV出力**（→ **マネーフォワードクラウド**へインポート）
- **管理画面は自宅からも確認**（クラウド前提の遠隔閲覧）

> 検討（クロエ）：①**配信は法対応が前提**＝広告メールは**特定電子メール法**でオプトイン取得・送信者表示・配信停止導線が必須。LINE配信は**LINE公式アカウント＋Messaging API**（無料通数に上限、超過は有料）。個人情報の取得同意・管理も要設計。②スタッフに数字を見せるのは効果的だが**自分の担当分のみ**に絞る。③ステップメール/LINEは**第3段**（開店必須ではない）。

### E. 集客チャネル（2026-07-02 確定：自作Webサイトは作らない）

**方針**：Yutaka 判断＝**宣伝用Webサイト・Zoho SalesIQ は不採用**。**LINE公式アカウント＋Facebook＋Instagram** の3つで集客・情報発信・再来店を回す。→ 集客は統合Web App の外＝**アプリの構築範囲が縮む（A〜Dの内部業務に集中でき、9/19に有利）**。

**2層のチャネル設計：**

| 層 | チャネル | 役割 |
|---|---|---|
| **発見・リーチ** | **Facebook／Instagram** | 在日インドネシアコミュニティ・研修生・過去常連・dangdut層に届く。イベント告知・写真/動画・拡散。投稿はインドネシア語中心＋日本語 |
| **受け皿・再来店** | **LINE公式アカウント** | 友だち追加を受け皿に、**イベント案内・臨時休業・クーポン・あいさつ/ステップ配信**。＝旧「宣伝サイト＋ステップメール」の役割をLINEが一手に担う |

- **集客ループ**：`FB/IG で発見 → LINE友だち追加（店頭QR・イベント・投稿から）→ LINEで情報配信/クーポン → 来店 → 伝票で来店/売上を記録 → 担当スタッフCRM(D)`。
- **効果測定**：LINE友だち数・FB/IGインサイト＋**店頭/イベントのQRとクーポン**（「LINE登録で1杯」等）で web→来店の貢献を数値化。SalesIQのような訪問トラッキングは不要。
- **CRMとの関係**：顧客DB・担当スタッフ紐付けは統合アプリ(D)が保持。配信はLINE公式で実施（将来必要なら Messaging API 連携を検討、当面は手動運用で十分）。
- **時期**：**LINE公式・FB・IGは開店前（9月前半）に開設**し、誕生日イベントの告知に使う。運用はカリサ／スタッフ（インドネシア語ネイティブが強み）。

**既存の集客資産（★開店ダッシュの起爆剤）：**

- **Peatix フォロワー 276人**：Yutaka が以前、インドネシアの歌手グループ **Walli Band** の大阪・名古屋ライブのチケット販売を手伝った際に付いたフォロワー。**Peatix から一斉送信できる**（※メールアドレスは非表示＝直接エクスポート/自前CRM取込は不可）。
  - **価値**：276人は「名古屋・大阪でインドネシア音楽ライブに来る層」＝**在日インドネシアコミュニティのど真ん中**。開店＝地元帰還の物語と完全に一致。名古屋在住分は直接来店見込み。
  - **使い方のコツ（クロエ）**：Peatixは本来**イベント告知の場**。生の「バー宣伝」を送ると規約・スパム感でフォロワーの信頼を損なう恐れ。→ **9/19開店を「イベント」として立てて告知する**のが正解：
    - 例）Peatix で**オープニング/誕生日イベント（＋できればdangdut/インドネシア音楽ナイト）を無料RSVP or 少額チケットで作成 → 276人へ一斉送信**。
    - 目的は**来店＋LINE友だち化**：イベント来場者を**店頭QRでLINE公式へ登録**させ、以後はLINEで継続接触（Peatix＝入口、LINE＝定着）。
  - **将来**：Walli Band 級のアーティストを呼ぶ夜＝この店の差別化（母国の音楽・生ライブ）にもなり得る。Peatixフォロワーはその集客母体。

- **インドネシア音楽シーンとの人的つながり（社会的証明・差別化）**：
  - **ST12（インドネシアの有名バンド）**：2026年6月の豊田スタジアムのイベントで来日→メンバーの**打ち上げにYutakaが参加**（焼き鳥→インドネシアバーREVELでST12やインドネシア語の歌を一緒に熱唱、意気投合）。**写真あり**。
  - **Walli Band** との写真もあり。
  - **活かし方**：店内に写真を掲示＝「本物のインドネシア文化・音楽とつながる店」という**コミュニティ内の信用と差別化**（過去の高齢日本人常連型では出せない、若年インドネシア層に刺さる強み）。
  - REVEL 等**名古屋のインドネシア系店との横のつながり**＝共催イベント・相互送客の余地（名古屋のインドネシア系は全店知り合いで協業予定＝非競合の無料マーケチャネル）。
  - **留意（クロエ）**：私的な記念写真の店内掲示は通常問題ないが、**SNS/宣伝で著名人の写真・名前を使う場合は本人の同意（肖像権・パブリシティ権）を取る**こと。同意が取れれば、開店に向けた**一言応援メッセージ／ショート動画**をもらえると Peatix・SNS の告知が跳ねる。

### E-2. LINE友だち獲得の設計（来店時に登録してもらう）（2026-07-02）

**原則**：無料特典は**原価ゼロ〜激安のものだけ**（カラオケ・ソフトドリンク・おばんざい1品）。**利益の柱＝酒は割引しない**。狙いは一時割引でなく**再来店（LTV）**。

- **きっかけ（その場で登録させる導線）**：卓上QRポップ（インドネシア語＋日本語）／**会計時にスタッフが一声「LINE登録で◯◯」**（会計動線に組込む＝登録率が上がる）／登録は**その場で特典即時提供**（即時性が登録率を最大化）。
- **登録特典（即時・低原価）候補**：①**カラオケ1曲/最初の30分無料**（原価ほぼ0・体験価値大＝一番おすすめ）②インドネシアのソフトドリンク（teh/kopi）1杯（原価激安・文化的に合う）③おばんざい1品サービス（原価率20〜28%＝低コスト高満足）。※ウェルカムのビール/サワー1杯は可だが酒は基本セット誘導へ。
- **再来店の仕掛け（本命）**：**LINE公式の「ショップカード」機能（無料）でデジタルスタンプカード**。**誕生日クーポン**（顧客誕生日にLINE配信→記念日消費は単価高い）。
  - **【重要・特典は客層に合わせる（2026-07-02 Yutaka指摘）】** 来店回数の多い**おばんざい客に「満了でボトル」はミスマッチ**（コストは小＝ボトル原価≈1,000円だが、酒を飲まない層には刺さらず、伸ばしたい行動＝飲む/滞在/人数が育たない）。→ **特典をセグメント別に設計**：
    - **食事リピーター向け**：おばんざい食べ放題1名無料／**カラオケ無料**／**次回“同伴1名無料”（連れは通常料金）**＝滞在・人数を増やし**自然に酒が出る方向**へ誘導（原価小・行動を育てる）。
    - **飲む客向け（夜）**：**ボトルキープ・ロイヤルティ**（例：ボトル◯本目を1本サービス）＝ボトルは原価激安で高満足、行動と一致。
  - **【金額ベーススタンプは不採用（2026-07-02 Yutaka指摘）】** 「¥3,000ごとに1スタンプ」は破綻：3万円のシャンパン1本で10スタンプ＝**1回でカード終了**＝“通う”動機が消える。**根因＝スタンプカードは"頻度"の道具で"金額"の道具ではない**。→ **飲む客はスタンプカードに乗せない**。
  - **確定＝客層で道具を変える2トラック**：
    - **頻度トラック（食事/一般）＝来店回数スタンプ**（1来店＝1スタンプ・金額無関係、満了でカラオケ/同伴1名/食べ放題）。LINEショップカードが得意な領域。シンプル。
    - **金額トラック（飲む客）＝スタンプではなく (a)ボトルキープ本数（◯本目サービス） (b)累計金額のVIPランク**（期間累計で特典・待遇）。カードを“埋める”発想をやめ、**伝票→CRM(D)で本数・累計を自動集計**（LINEカードではなくアプリ側で管理）。
    - **高額客（例：3万シャンパン級）は特典で釣らず“VIP待遇・関係”で維持**：ボトルキープ棚・指名スタッフのケア・記念日ドンペリ等＝ステータスと関係性がリテンション。
  - 要は**「食事客＝回数スタンプで“また来る”、飲む客＝本数/VIPで“たくさん飲む・特別扱い”」**。2フェーズ業態に報酬も2系統。
- **開店イベント連動**：Peatix来場者は**当日LINE登録で開店記念特典/抽選**（ST12・Walli写真ポストカード、dangdutリクエスト権 等＝原価小・話題性大）。
- **★手元の現物ノベルティ（2026-07-02 Yutaka）＝登録フックに最適**：
  - **teh kotak（インドネシアの箱入り紅茶）**：安価・万人向け（酒を飲まない層にも刺さる）・母国の味＝**日常の「LINE登録でその場プレゼント」定番**。
  - **Walli Tシャツ 数百枚**（昨年の販売残を無償入手）：**原価ゼロ・高知覚価値・音楽ファンアイテム**。着て歩けば**インドネシアコミュニティ内の歩く広告**。→ **開店ダッシュの限定フックに集中投下**：「9/19開店〜期間限定/先着、**LINE登録＆来店でWalli Tシャツ進呈**」。恒常配布で在庫を溶かさず、**launch期に友だち数を一気に積む**のに使う。サイズ差は「無くなり次第終了」で処理。
  - 使い分け：**teh kotak＝恒常の即時フック／Tシャツ＝開店期の限定フック**。どちらも**登録（獲得）用**で、再来店の**ショップカードとは役割を分ける**。
  - 留意：Walli関連は関係性があるので、宣伝で名を出す際は**Walli側へ一声（礼儀＋クロスプロモの芽）**。物品配布自体は所有物で問題なし。
- **スタッフ連動**：スタッフが「私の担当客として登録して」と声かけ→**顧客に担当スタッフ紐付け(D)**＝スタッフ自身の客リストになりモチベ＆指名文化と整合。
- **留意**：LINE公式の一斉配信は**無料枠に通数上限**（増えたら有料プラン）＝配信は乱発せず重要告知/セグメントに絞る。特典で"クレクレ客"ばかり集めず、コミュニティのたまり場価値で来る客を中心に。

## 4. 非機能・前提

| 項目 | 内容 |
|---|---|
| 端末 | iPad（店頭オペ）／PC・スマホ（自宅から管理画面） |
| プリンタ | mC-Print3（Webからの印刷疎通を最初に確認・型番も確認） |
| 稼働形態 | クラウド（自宅から閲覧）。店の回線断時の会計継続（オフライン耐性）は方針を決める |
| 言語 | 日本語＋英語＋インドネシア語（メニュー中心） |
| 会計連携 | MFクラウドのCSV取込フォーマットに列設計を合わせる |
| 対象外（当面） | 賞与・日雇丙欄・年末調整・住民税・雇用保険料の計算（Excel注記どおり） |

## 5. リリース計画（8月開店を落とさない段階リリース）

| 段階 | 範囲 | 目的 | 時期 |
|---|---|---|---|
| **MVP** | A：iPad伝票→会計→mC-Print3レシート→**スタッフ紐付け**→日次売上→CSV | 店が回る＋誰の売上か記録される | **8月・開店必須** |
| **第2段** | C：スタッフ別売上→歩合→源泉（Excel移植）→**月2回給与** | 給与計算を自動化（初回給与に間に合えば可・それまではExcelで手計算の保険あり） | 開店直後 |
| **第3段** | B：出退勤打刻／D：顧客×スタッフ紐付け・スタッフ向け成績ダッシュボード・**ステップメール/LINE配信**・多言語UIの拡充 | 運用を楽に・再来店を作る | 開店後 |
| **並行(集客)** | E：**LINE公式＋Facebook＋Instagram を開設・運用**（自作サイトなし）。店頭QRでLINE友だち獲得 | 開店告知・オプトイン獲得・再来店 | **9月前半に開設** |

> フル機能を8月に一度に作るのはリスク。**MVPで開店を確実にし、給与・勤怠・顧客は段階で載せる**。既存Excelがあるので給与は手計算で繋げる＝第2段が多少ずれても実害小。

## 6. 着手前に決める/確認する（判断待ち）

1. **雇用 or 業務委託**：Excelは給与所得(月額表)前提＝**雇用**。出退勤・ベース賃金とも整合。雇用なら労働時間管理・最低賃金・(条件次第で)社保/労災の義務。→ **税理士に区分確認**。
2. **月2回払いの源泉の当て方**（合算か各回か）を国税庁ルールで確認してから実装。
3. **甲/乙の適用**：掛け持ちスタッフは乙欄（扶養控除等申告書は1社のみ甲）。
4. **技術**：mC-Print3のWeb印刷疎通、オフライン耐性の要否。
5. **作るか/ハイブリッドか**：フル自作 or 「エアレジ＋自作の歩合/源泉ツール」の最終判断（[requirements-notes.md](requirements-notes.md) 参照）。自作なら本構想でMVP設計へ。
