# Business Data Accumulation Rules

## 目的

業務システムで発生するデータを、単なる画面表示用ではなく、業務改善、KPI確認、監査、AI分析、将来の経営判断に使える形で蓄積する。

## 基本方針

- 業務データは、入力・更新・削除・承認・通知・出力の履歴まで含めて設計する。
- 画面や機能を作る前に、どの業務データを蓄積するかを定義する。
- データは、誰が、いつ、どの業務で、何のために使うかを明確にする。
- KPIに使うデータは、集計条件、更新頻度、責任者を決める。
- AI分析に使う可能性があるデータは、元データ、構造化データ、メタ情報を分けて保存する。
- 個人情報、機密情報、顧客情報は保管目的、閲覧権限、保存期間を定義する。
- 削除ではなく履歴保持が必要な業務は、論理削除または監査ログを設計する。

## 蓄積すべきデータ分類

| 分類 | 内容 | 例 |
|---|---|---|
| Master Data | 業務の基準情報 | 顧客、商品、社員、部署、権限、工程 |
| Transaction Data | 日々発生する業務データ | 案件、見積、受注、作業、検査、問い合わせ |
| Status Data | 業務状態 | 未対応、対応中、承認待ち、完了、差戻し |
| Activity Log | 操作履歴 | 登録、更新、削除、承認、出力、ログイン |
| Comment / Note | 判断・補足情報 | 社内メモ、対応履歴、申し送り |
| Attachment | 関連ファイル | 図面、写真、PDF、伝票、契約書 |
| KPI Data | 集計用データ | 売上、粗利、工数、リードタイム、不良率 |
| AI Source Data | AI分析元データ | 問い合わせ、議事録、作業記録、ナレッジ |
| Derived Data | 派生データ | 集計結果、スコア、分類、要約、AI出力 |

## 設計時の必須確認

- このデータは何の業務で発生するか
- 誰が入力するか
- 誰が確認するか
- 誰が編集できるか
- 削除してよいか、履歴保持が必要か
- KPIに使うか
- AI分析に使うか
- 外部出力するか
- 保存期間は必要か
- 個人情報・機密情報を含むか

## DB設計ルール

- 主要テーブルには `created_at`, `updated_at` を持たせる。
- 業務上削除履歴が必要なものは `deleted_at` または監査ログを設計する。
- ステータス項目は定義表またはEnum設計に記録する。
- 金額、数量、日付、担当者、ステータスは型と単位を明確にする。
- 集計に使う項目はIndexを検討する。
- 添付ファイルはDBに直接バイナリ保存せず、保存先とメタデータを分ける。
- AI出力は元データと分け、再生成可能性と生成日時を記録する。

## 監査ログルール

以下の操作は監査対象にする。

- 重要データの作成
- 重要データの更新
- 重要データの削除
- 承認、差戻し、取消
- 権限変更
- CSV出力
- ファイルダウンロード
- 外部連携実行
- AIによる分類、要約、判定

## AI分析用データルール

- 元データを保存する場合は、出典、取得日、作成者、対象業務を記録する。
- AIが生成した要約、分類、スコアは派生データとして扱う。
- 派生データにはモデル名、プロンプト版数、生成日時を記録する。
- AI出力を業務判断に使う場合は、人による確認フラグを持たせる。
- 個人情報を含むデータは、AI投入可否を明示する。

## 禁止

- 画面で必要な項目だけをDB化して終わること
- 履歴が必要な業務で上書きのみの設計にすること
- KPI定義なしにダッシュボードを作ること
- AI分析元データとAI出力を同じ意味で扱うこと
- 個人情報を分類せずに蓄積すること
