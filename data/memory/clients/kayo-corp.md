## 課題・ニーズ（事実/仮説）

- テーマ＝**DX・業務効率化**。具体＝**三菱電機向け原価の"部品構成ファイル(Excel多階層BOM)"の集計・DB化・分析**。
- あるべき姿：①各部品構成ファイル(1ファイル=1ASSY、A/B2レイアウト・可変行)を Level付きで再帰結合 ②年間支給部材一覧と親子結合 ③Python で SQLite(parts_data.db)へ抽出 ④Power Query の ODBC 接続で現場が検索・分析。要件詳細 [work/kayo-corp/01_business/部材表DB化_要件整理.md](../../../work/kayo-corp/01_business/部材表DB化_要件整理.md)。
- 【判断】これは**属人化した原価Excelの脱・属人化＋分析基盤化**。三菱依存下で利益を守るには原価の見える化が効く＝経営効果が明快。会計士の関心（原価・利益）とも直結。

## 案件経緯（契約書より確定・2026-07-01）

- **前回契約＝2026/2/4〜2/28・20万円(税別)・準委任**。YSSとして受託。契約サマリ [work/kayo-corp/03_documents/契約サマリ.md](../../../work/kayo-corp/03_documents/契約サマリ.md)。
- **対象＝Excel部材表(BOM)**。やったこと＝①現状把握・整理 ②課題抽出 ③簡素化・業務効率化の改善案 ④**システム化の提案資料作成**まで。
- **契約で実装・開発・運用保守は明示除外**＝前回は上流（診断・提案）フェーズ。
- 【再提案の種】「対応しきれなかったこと」＝**実装・ツール化/DB化・運用**。最初から次フェーズに切り出されていた領域で、負け筋ではない。
- **結末（2026-07-01判明）：提案は止まった。理由＝Excelシートに保存されたBOMと三菱の支給品シートの"照合"をシステム化するのが技術的に困難だったため**。前回は SQLite＋Power Query の路線で試作(参考スクリプト群 `04_system/202602/`)まで作ったが、BOM×支給の突合が壁になった。
- **今回の再提案の方向（Yutaka意向）：Web App で BOMファイルと三菱支給ファイルを照合し、①粗利 ②支給部品比率 ③"支給→自社調達"に切り替えた場合の粗利改善シミュレーション を可視化するシステム**。コンセプト [work/kayo-corp/02_proposal/再提案コンセプト_WebApp.md](../../../work/kayo-corp/02_proposal/再提案コンセプト_WebApp.md)。
- 【クロエの反証】前回を殺したのは"照合"。**Web化はUI/提供形態の改善であって、照合ロジックの困難そのものは解けない**。再提案の成否は「BOMの部品コードと支給明細の突合を、例外(親なし単独シート・A/B可変レイアウト・生産終了品)込みで安定処理できるか」に懸かる。ここを実データ9件＋支給明細で**先にPoC検証してから提案**するのが定石。UIから入ると同じ壁で二度死ぬ。
